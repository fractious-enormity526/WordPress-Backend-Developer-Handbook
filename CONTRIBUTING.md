# Contributing

First of all, thank you for taking the time to look at this project.

This handbook started as a personal learning resource and has gradually grown into something I thought might be useful to other WordPress developers as well.

Although I currently maintain the project on my own and in my spare time, I genuinely appreciate bug reports, corrections, suggestions and improvements.

Every issue opened, typo fixed or idea shared helps make the handbook a little better.

---

# Before You Contribute

Please remember what this project is trying to achieve.

The goal is **not** to replace the official WordPress documentation.

Instead, the goal is to:

* organize information into a structured handbook
* provide an offline reference
* improve discoverability
* make the material easier to study
* work well with tools such as Google NotebookLM

Whenever possible, I'd like to stay as close as possible to the original documentation while improving readability.

---

# Ways You Can Help

There are many ways to contribute besides writing code.

For example:

## 📝 Correct Typographical Errors

Found a typo?

Please let me know.

Even small improvements make the handbook nicer to read.

---

## 📖 Improve Formatting

Formatting is currently one of the biggest areas for improvement.

Examples include:

* spacing
* page breaks
* headings
* code blocks
* tables
* images
* typography

These improvements are always appreciated.

---

## 🔗 Report Broken Links

Documentation changes over time.

If you notice:

* outdated URLs
* moved documentation
* broken references

please open an issue.

---

## 📚 Suggest Missing Topics

If there's a topic that you think belongs in the handbook, I'd love to hear about it.

Examples might include:

* WooCommerce
* React
* Gutenberg
* Security
* Performance
* Headless WordPress

or something I simply overlooked.

---

## 🐞 Report Errors

Nobody writes 4,800 pages without making mistakes.

If you notice:

* incorrect information
* outdated examples
* confusing explanations

please let me know.

---

# Pull Requests

Pull requests are welcome.

To make reviewing easier, please try to keep them focused.

Good examples include:

* fixing one issue
* improving formatting
* updating documentation
* correcting examples
* improving scripts

Large changes are also welcome, but it may be helpful to discuss them first by opening an issue.

---

# Coding Style

If you're contributing to the scripts used to generate the handbook:

* keep the code readable
* prefer simplicity over cleverness
* document non-obvious logic
* avoid unnecessary dependencies where possible

Consistency is generally more important than personal preference.

---

# Documentation Style

When suggesting documentation changes, I try to follow a few simple principles.

## Be Practical

Prefer real examples over abstract explanations.

---

## Be Accurate

When possible, align with the official documentation.

---

## Be Beginner-Friendly

Complex topics should be approachable without becoming oversimplified.

---

## Keep It Searchable

Clear headings and logical structure make both PDFs and NotebookLM much easier to use.

---

# Issues

If you're opening an issue, it helps if you include:

* a clear description
* the affected chapter
* page number (if applicable)
* screenshots (if formatting is involved)
* steps to reproduce (for script issues)

The more context you provide, the easier it is to investigate.

---

# Feature Requests

Feature requests are always welcome.

Please understand that I work on this project in my spare time, so I can't promise every suggestion will be implemented.

That said, I keep track of good ideas and many of them eventually make their way into the roadmap.

---

# Development Priorities

Right now I'm primarily focusing on:

* improving PDF formatting
* fixing documentation issues
* expanding NotebookLM support
* React for WordPress Developers
* Gutenberg Block Development

Beyond that, I'm always open to ideas.

---

# Be Respectful

Constructive criticism is always appreciated.

Everyone has different experiences and perspectives, and respectful discussions often lead to better documentation.

Please assume good intentions and keep feedback friendly and constructive.

---

# Thank You

Whether you've reported a typo, suggested a new topic, improved a scraping script or simply starred the repository, thank you.

Knowing that other developers find this handbook useful is what motivates me to continue improving it.

Happy coding! 🚀
